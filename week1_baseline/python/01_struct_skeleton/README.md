# 01 · Struct Skeleton (Python)

## Setup

This step shares the single virtualenv at the **repo root**:

```bash
# from the repo root
python3 -m venv .venv
source .venv/bin/activate
pip install -r week1_baseline/python/01_struct_skeleton/requirements.txt
```

The launcher at `week1_baseline/bin/python/01_struct_skeleton` assumes `.venv`
already exists at the repo root and has these dependencies installed.

---

This step keeps the configuration work from step 0 and adds lightweight data
structures for values passed through the agent loop:

- `boukensha.Tool`
- `boukensha.Message`
- `boukensha.Context`

## Design Considerations

Ruby uses `Struct.new` for these records. The Python port uses dataclasses for
`Tool` and `Message`, plus a small `Context` class so each context gets its own
message history and tool registry.

The display strings intentionally keep the Ruby-style shape (`#<Tool ...>`,
`#<Message ...>`, `#<Context ...>`) so snapshots stay easy to compare with the
Ruby baseline.

| File | Purpose |

## Data Structures

### `boukensha.Tool`

| Field | Description |
|---|---|
| `name` | The name of the tool |
| `description` | Shown to the agent so it knows when to invoke the tool |
| `parameters` | The arguments that need to be passed in |
| `block` | The callable that will eventually run when the tool is called |

Example:

```
#<Tool name=move description=Move the player in a direction (north, sout params=['direction']>
```

### `boukensha.Message`

| Field | Description |
|---|---|
| `role` | Who is speaking: user, assistant, or tool_result |
| `content` | Text generated by the agent or provided by the user |
| `tool_use_id` | Links a tool result back to the specific tool call |

Examples:

```
#<Message role=user content=Explore north and tell me what you find....>
#<Message role=assistant content=Sure, let me head north and take a look....>
#<Message role=tool_result [toolu_01X] content=You move north into a torch-lit corridor....>
```

### `boukensha.Context`

| Field | Description |
|---|---|
| `task` | The task class that owns the context |
| `system` | System prompt sent separately from the message history |
| `messages` | Full conversation history |
| `tools` | Registered tools available to the agent |

Example:

```
#<Context task=player turns=2 tools=1>
```
